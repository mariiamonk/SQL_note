# Глава 1. Введение и Архитектура СУБД

### 1.1. Данные vs Информация

  * **Данные (Data):** Сырые значения без контекста.
  * **Факт (Fact):** Данные с привязанной семантикой (\"01.01 — дата отгрузки").
  * **Информация (Information):** Обработанные факты, пригодные для принятия решений ("Прибыль за январь выросла на 10%").
  * **СУБД (DBMS):** Программное обеспечение для управления данными, обеспечивающее переход от хранения сырых байтов к получению информации.

### 1.2. Недостатки файловых систем

Использование обычных файлов (CSV, Excel) для хранения данных критично из-за:

1.  **Дублирования (Redundancy):** Одно и то же значение хранится в разных местах, что ведет к росту объема.
2.  **Аномалий (Anomalies):** \* *Обновления:* Изменили адрес в одном файле, но забыли в другом.
      * *Удаления:* При удалении записи о заказе стирается информация о клиенте.
3.  **Отсутствия связности:** Нет механизма, гарантирующего, что FK (внешний ключ) ссылается на существующий PK (первичный ключ).
4.  **Целостности:** Сложно реализовать одновременный доступ для 1000+ пользователей без порчи данных.

### 1.3. Физическое хранение и Buffer Pool

СУБД оптимизируют работу с диском (самым медленным узлом):

  * **Страницы (Pages):** Минимальная единица обмена данными между диском и памятью (обычно 8 КБ).
  * **Buffer Pool:** Область в RAM, где СУБД кэширует прочитанные с диска страницы.
  * **Принцип:** Сначала поиск в Buffer Pool. Если страницы там нет — чтение с диска (Disk I/O). Запись также сначала фиксируется в памяти и логе (WAL), а затем сбрасывается на диск.

### 1.4. Форматы хранения: Row, Column и Parquet

Выбор формата зависит от типа нагрузки:

| Формат | Тип (Store) | Особенности | Применение |
| :--- | :--- | :--- | :--- |
| **Row (Строковый)** | OLTP (Postgres) | Строка записывается целиком. Быстрая вставка и поиск одной записи. | Транзакции, веб-сервисы. |
| **Column (Колоночный)** | OLAP (ClickHouse) | Данные одной колонки лежат рядом. Высокое сжатие. | Аналитика, отчеты. |
| **Parquet** | Колоночный (Файловый) | Гибридный подход. Данные разбиты на группы строк (Row Groups), внутри которых упакованы по колонкам. | Big Data, Data Science (Hadoop/S3). |

#### Почему важен Parquet?

**Apache Parquet** — это открытый бинарный формат. Его эффективность обусловлена:

  * **Пропуском данных (Predicate Pushdown):** Если запрос ищет `price > 1000`, СУБД читает метаданные и сразу пропускает группы строк, где макс. цена меньше 1000.
  * **Сжатием:** Одинаковые типы данных (например, целые числа в колонке возраста) сжимаются гораздо лучше, чем разнородные строки.
  * **Схемой:** Описание структуры хранится в самом файле.
